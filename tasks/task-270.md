### Task 270: Turn the post-reveal intent capture into an inline conversation thread

Implemented and verified: `src/lib/contact-intent-logic.ts` (new, unit-tested), `src/app/api/contact-intent/route.ts`, `src/components/contact-intent-form.tsx`, `src/lib/types.ts`, plus e2e coverage across the four `e2e/contact-form-*.spec.ts` files and a new `e2e/contact-form-intent-thread.spec.ts`.

Follows directly from task-269 (`src/components/contact-intent-form.tsx`, `src/app/api/contact-intent/route.ts`), which shipped a single-shot optional capture: one textarea, one debounced auto-submit, one freeform thank-you reply. Mannan now wants that turned into a lightweight back-and-forth conversation that still visually looks like one plain input box.

#### What Mannan asked for (verbatim intent, cleaned up)
- [x] Remove the separate "(Optional) Share what brought you here today" header line — folded into the textarea's placeholder (`Your name, and/or why you're here`, shown only before the first turn).
- [x] Drop the current debounce/staleness-guard machinery. `latestTextRef`/`inFlightTextRef` are gone (turn-locking makes them structurally unnecessary); `pendingSinceRef`+`MAX_PENDING_MS` stayed as a bounded-wait ceiling (see the correction in the resolved-questions section below) since it protects a real, still-applicable mobile bug fix.
- [x] Loosen the LLM's job to open-ended recognition — the fixed 5-category classifier tool is gone entirely (nothing consumed it downstream anyway); the model just replies naturally and tags nothing.
- [x] After a reply comes back, the user's message + the AI's reply become view-only, stacked in order, inside the same box (`contact-intent-turn` / `-turn-user` / `-turn-ai`).
- [x] A new editable area appears below for the next turn, and can answer a follow-up question there.
- [x] Loading indicator carries forward — reskinned as a cycling Braille-dots spinner (terminal-appropriate) but still bottom-right of the input.
- [x] Visually still one plain input box — the border/radius/background/padding that used to live on the `<textarea>` now lives on the wrapping `.thread-box` div; with zero turns sent it's pixel-identical to before. (Superseded in one respect: Mannan asked mid-implementation for a terminal-transcript look — see the visual-treatment decision below — so the *font* changed from sans to monospace and a `>` prompt was added; the box's own geometry did not change.)

#### Why this needs design thought before code (the hard part)
A native `<textarea>` cannot have mixed read-only/editable regions — its content is one flat, fully-editable string. "View-only past turns + editable latest turn, all inside what looks like a single textarea" cannot be built with a real `<textarea>` alone. This needs a look-alike custom component: a bordered container styled identically to today's textarea, containing (a) rendered, non-editable past turns (plain divs/paragraphs, not form elements) stacked in scroll order, and (b) a real, small, unstyled-border textarea or content-editable line pinned at the bottom for the active turn — visually seamless, structurally two different things. Get the container styling (border, radius, background, padding, font, scroll behavior) pixel-identical to what's live today so nothing about the box itself appears to change.

#### Open questions to resolve with Mannan before implementing — RESOLVED
Explored via an interactive HTML/CSS/JS artifact prototype (real debounce timing, mock keyword-based LLM, live turn cap and one-question logic) before touching real code. Mannan reviewed it and asked for the terminal-transcript visual direction specifically; decisions below are confirmed, not just proposed.

- [x] Submit trigger for each turn: **automatic pause-to-send stays the default** (900ms after typing stops, no button). Enter optionally fast-tracks a send immediately; Shift+Enter still inserts a newline. `latestTextRef`/`inFlightTextRef` (in-flight staleness guards) are dropped, since turn-locking makes overlapping requests for the same turn structurally impossible. **Correction from the design spike:** `MAX_PENDING_MS` itself is *not* fully dropped — it's kept as a bounded-wait ceiling (`pendingSinceRef` + a 3s cap), because it's what makes `e2e/contact-form-mobile.spec.ts`'s soft-keyboard-storm tests pass (some mobile keyboards fire continuous synthetic input events that would otherwise reset a plain debounce forever and the message would never send). That guarantee is orthogonal to single-shot vs. turn-based and isn't safe to drop.
- [x] Visual treatment of past turns: **terminal transcript**, not plain stacked text or "You:"/"Mannan:" labels. A `>` prompt (blue) prefixes what the user sent, with a CSS hanging indent so wrapped lines align under the text, not under the `>`. The AI's reply renders unprefixed in green (reusing the existing "AI reply" green from today's `done` state) — authentic terminal stdout convention (only the input line gets a prompt). Monospace font throughout the box.
- [x] Reopen persistence: **does not persist.** Every modal reopen starts a fresh thread, matching today's behavior where `CLOSE_CONTACT_MODAL` already discards `contactResult`/`contactShowResult`. No new AppContext or sessionStorage plumbing for what's meant to be a quick aside, not a saved conversation.
- [x] Turn limit & rate limit: **hard cap at 3 user turns**; the existing 10/hour/IP limiter is left untouched (3 turns caps a single visit at ≤4 requests, well inside budget). The thread ends by quietly not offering a new input line — no "conversation ended" message.
- [x] Follow-up questions: **explicit, capped feature** — the model may ask exactly one clarifying question, ever, per thread (not one per turn), and only when it would concretely help Mannan act (e.g. a job mention with no company named). After that one use, later turns may only acknowledge. This is enforced by the system prompt *and* defensively re-checked server-side (a trailing "?" is stripped if the model tries to ask again after already using its one question).

#### Lessons from task-269 this build must not repeat
- [x] Every tracked status renders something real: idle (empty box), sending (Braille spinner bottom-right), error (red text below box + turn reverted to editable), and the terminal history itself. Caught and fixed one live instance of this exact lesson during screenshot verification — the placeholder text was reappearing during the very first turn's sending window, which read as "type this again?"; fixed by gating it on `pendingText === null` too.
- [x] System prompt (`src/app/api/contact-intent/route.ts`) defaults to a plain thank-you and only allows a clarifying question when genuinely useful, capped at one per thread — enforced in the prompt and defensively re-checked server-side in `normalizeResult`.
- [x] Screenshot-verified against the real running app (not just the design-spike artifact) via Playwright: idle/placeholder, first completed turn, multi-turn history persisting, sending (spinner visible mid-flight), follow-up-question turn, 3-turn cap ending, and error-revert-to-editable. Screenshots live under `e2e/screenshots/`.
