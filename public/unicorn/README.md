# Unicorn Studio scene ‚Äî `health-hero-scene`

The hero animation on `/garden/article/health-longevity` ("Health is an Artform") is rendered by [Unicorn Studio](https://unicorn.studio) from a compiled JSON export.

## Two files, one source of truth

| File | Role | Hand-edit? |
|---|---|---|
| `health-hero-scene.raw.json` | **Pristine** export from Unicorn Studio. Source of truth. | Never. Replace wholesale on re-export. |
| `health-hero-scene.json` | **Derived** file loaded by the app. Generated by the transform script. | Never. It is overwritten on every build. |

The app loads `health-hero-scene.json` via `src/components/garden/garden-hero.tsx` ‚Üí `UnicornScene`. You should not edit that file directly; your edits will be wiped.

## Transforms applied on top of the raw export

The derivation script is `scripts/apply-unicorn-transforms.mjs`. It currently does two things:

1. **Speed √∑ 3** ‚Äî every layer with a numeric `speed` field is slowed 3√ó so the animation feels meditative rather than frantic.
2. **Iridescent recolor** ‚Äî the baked-in orange constant `vec3(1, 0.537, 0.302)` inside the `wisps` and `outline` layers' compiled fragment shaders is replaced with a time + position driven cosine palette. The `wisps` layer gets an animated iridescence (it has `uTime`); `outline` gets a static spatial iridescence (no `uTime` uniform).

## Re-export workflow

Whenever you tweak the scene in Unicorn Studio and need to bring the new design into the repo:

1. Export from Unicorn Studio.
2. Replace `health-hero-scene.raw.json` with the new export.
3. Run `bun run unicorn:build` (or just `bun install` ‚Äî the `postinstall` hook re-derives it automatically).
4. Inspect `/garden/article/health-longevity` locally.
5. Commit **both** files together.

The derivation is idempotent and deterministic, so any clone of the repo produces the same final scene. The `postinstall` hook guarantees the scene is derived on any fresh `bun install` without the developer having to remember.

## Why this dance exists ‚Äî the three options we considered

Unicorn Studio exports bake colors and animation parameters into a single JSON that contains compiled GLSL shader strings. The orange tint is not an exposed knob ‚Äî it lives inside those shader strings. When we want the scene to look different from what the editor produces, we have to choose between three approaches:

**Option A ‚Äî post-export transform script (what this repo uses).** Keep the pristine export, derive the final JSON from it via a committed script. Pros: version-controlled, reproducible across clones, survives re-exports. Cons: the patches are string replacements on compiled GLSL ‚Äî if Unicorn Studio changes how they compile shaders in a future version, the regexes may silently stop matching. The script logs how many replacements it made; watch that number on re-export.

**Option B ‚Äî document only.** Maintain a README (this one, sort of) warning "do not re-export without re-applying the patch." Pros: zero code. Cons: brittle; relies on humans remembering.

**Option C ‚Äî wrapper shader outside the JSON.** Render Unicorn Studio's canvas underneath our own WebGL canvas and apply iridescence as a second-pass shader in app code. Pros: completely decoupled from Unicorn Studio's export format ‚Äî re-exports can never break it. Cons: significantly more code, extra GPU cost, and duplicates the canvas layer.

We went with A + B. If the shader-string replacements in the script ever start failing silently (e.g. Unicorn Studio changes their GLSL codegen), the right move is probably **option C**: move the iridescent tinting into app code and let Unicorn Studio own only the "raw" look.

## Extending the transform

If you want to add new post-export transforms (e.g. a different palette, a new hover behavior, resolution tweaks), add them to `scripts/apply-unicorn-transforms.mjs`. Keep each transform small, targeted, and logged so regressions are visible.
